# Internationalization (i18n)

**Status:** DRAFT  
**Last updated:** 2026-05-29

This spec defines how page text is translated by runtime code in `src/ui/scripts/i18n` and locale files under `src/public/data/locales/`.

The module is built for static landing pages and small multi-page sites. It is not a full app-level i18n framework.

## Goals

- Keep page readable before JavaScript and locale files load.
- Keep page readable when locale loading fails.
- Avoid fetching default-locale data unless needed for recovery.
- Keep custom elements responsible for their own text.
- Catch locale drift at build time instead of adding complex runtime fallback.

## Core Model

Each translated element uses `data-i18n`:

```html
<h1 data-i18n="home.hero.headline">This is a headline</h1>
```

The element's original literal text is its fallback text. Runtime translation may replace `textContent`, but fallback must stay available so a later missing translation can restore readable default text.

This means:

- Before i18n loads, browser shows default-language HTML.
- If active locale has translation, text is replaced.
- If active locale is missing a key, original fallback text is shown.
- If locale file fails to load, original fallback text remains or is restored.

There is no per-key fallback to another locale dictionary. The HTML fallback is the fallback.

## Translation Pass

Apps construct an i18n instance with an explicit config object:

```ts
export const i18n = new I18n({
  defaultLocale,
  locales,
  getLocaleFile,
  getLocaleDirection,
  isLocale,
});
```

`init()` resolves the starting locale, loads its dictionary, syncs `<html lang>` and `<html dir>`, then translates the bound root.

`setLocale()` loads the requested locale, updates state, syncs document locale, and re-runs the same translation pass.

Only elements under the bound root with `data-i18n` are considered.

### Leaf Elements Only

Translation writes to `textContent`, so it only applies to leaf elements. It does not translate attributes such as `aria-label`, `alt`, `title`, `placeholder`, or meta `content`.

If a `data-i18n` element has child elements, it is skipped and a warning is logged. Existing markup must not be destroyed.

### Custom Elements

Automatic page translation must not touch custom-element content. A custom element is any tag name containing `-`.

Default behavior:

- If the page root is `document`, the translator skips custom elements and their descendants.
- If the bound root is a custom element, translation is skipped.
- If the bound root is inside a custom element, translation is also skipped.

Custom elements translate themselves with `i18n.translate(key)` and their own fallback text:

```ts
title.textContent = i18n.translate("card.title") ?? "Default title";
```

Components must handle both lifecycle orders:

- If connected before i18n is ready, render fallback text and update on `ready`.
- If connected after i18n is ready, check `i18n.ready` and translate immediately.
- Always update again on `locale-change`.

Short, stable accessibility labels that describe a control instead of page copy may stay in the component code. This includes ARIA labels such as language menu labels and locale option alt text. Translate them only when the localized wording matters for the user experience or when the same text is already part of visible translated content.

## Locale Resolution

`LOCALE_MANIFEST` in `src/ui/scripts/i18n/manifest.ts` is the source of truth for supported locales. Each entry defines: `code`, `label`, `dir`, `icon`, and `file`.

Locale icons are declared for future locale selector UI. Files should live under `src/public/assets/icons/locales/` with the locale code as filename and `.webp` extension, for example `src/public/assets/icons/locales/en-US.webp`.

Declaration order is priority order.

Initial locale is chosen in this order:

1. Persisted locale from `localStorage`.
2. Browser locale from `navigator.languages`.
3. `DEFAULT_LOCALE`.

Browser matching rules:

- Exact locale match first, case-insensitive.
- Then language-subtag match, such as `pt` or `pt-PT` matching `pt-BR`.
- If several locales share a subtag, manifest order decides.

A locale is persisted only after that requested locale loads successfully. A failed request must not overwrite stored preference.

## Locale Files

A locale file is a flat JSON object:

```json
{
  "home.hero.headline": "This is a headline"
}
```

Rules:

- Keys are strings.
- Values are strings.
- Nested objects are invalid.
- Arrays are invalid.
- Empty objects are valid JSON shape but degraded content.

Runtime fetches only the active locale. It does not fetch the default locale for normal rendering because per-key dictionary fallback does not exist.

The default locale is fetched only as recovery when a non-default active locale cannot be loaded at all.

If active locale is already the default, only the default file is fetched.

## Failures And Recovery

Each locale fetch has a timeout. A stalled request is aborted so `init()` and `setLocale()` cannot hang forever.

Transient failures are retried with exponential backoff:

- timeout
- network error
- `408`
- `429`
- `5xx`

Permanent failures are not retried inside the same automatic resolution path:

- `404`
- malformed JSON
- invalid dictionary shape

Recovery is still allowed:

- Explicit `setLocale()` may retry a previously failed locale.
- `init()` may retry a previously failed locale.
- Empty locale files are not cached, so later `setLocale()` or `init()` can fetch them again.

This keeps recovery possible after deploy fixes, CDN repair, or temporary bad responses.

When every usable locale fails, translated elements show their original fallback text. The `ready` event reports `translationsAvailable: false` so consumers can show degraded-state UI if needed.

## Events

The i18n instance extends `EventTarget`.

`ready` fires once after `init()` resolves and applies initial state:

```text
{ locale, translationsAvailable }
```

`locale-change` fires after successful locale switch:

```text
{ locale, previousLocale }
```

`locale-change` does not fire when requested locale fails and fallback state is used instead.

## Build-Time Validation

The Vite plugin in `plugins/vite/i18n/` validates locale data during dev/build. `vite.config.ts` passes `LOCALE_MANIFEST` into the plugin.

It fails when:

- Locale files declared in `LOCALE_MANIFEST` are missing.
- Extra locale files exist but are not declared in `LOCALE_MANIFEST`.
- Locale files do not share the same key set.
- Locale file values are not strings.
- HTML uses a `data-i18n` key missing from locale files.

This prevents normal shipped builds from having missing locale files, mismatched locale dictionaries, or missing keys used by HTML `data-i18n` elements. Keys used from TypeScript, custom elements, feedback messages, or manually translated attributes are runtime-owned and should provide their own fallback text. Runtime fallback still exists for broken deploys, bad network, malformed responses, or other degraded states.

## Adding A Locale

1. Add entry to `LOCALE_MANIFEST`.
2. Add locale icon under `src/public/assets/icons/locales/` at the declared `icon` path.
3. Add JSON file under `src/public/data/locales/` at the declared `file` path.
4. Use exactly same keys as existing locale files.
5. Run build or validation through normal dev/build flow.
